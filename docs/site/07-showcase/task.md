## Objective

You have read about the task tree. Here you can open one and click around.

This page links to two live exports of real superRA task trees. They are the actual task-tracker interface, rendered by the same dashboard you get when you run superRA yourself, not screenshots or a simplified mock-up. The documentation site you are reading is itself a dashboard export, so the chrome around these trees is the chrome around this page. Everything you read about in the [Quickstart](#/02-quickstart) and the [task-tree skill](#/04-utility-skills) is something you can now see directly: the status of every task, how a parent's status rolls up from its children, the dependency graph, and the review conversation inside an individual task.

### What you are looking at

Each export is a self-contained page with the full task-tracker chrome. As you explore, these are the elements to notice — each one maps back to a concept page.

- **Status pills.** Every task carries a colored status: green for `approved`, yellow for `implemented` (done but not yet reviewed), red for `revise` (a reviewer sent it back), blue for `in-progress`, grey for `not-started`. The [status lifecycle](#/04-utility-skills/01-task-tree/03-status-and-frontier) explains how a task moves between them.
- **Rollup.** A parent task's status is computed from its children, not set by hand. A tree with one child in `revise` shows the parent as un-finished — you can see at a glance that the project is not done, and where the holdup is.
- **The DAG.** The dependency graph view draws the edges between tasks: which task must finish before which can start. A task with an unmet dependency is off the [frontier](#/04-utility-skills/01-task-tree/03-status-and-frontier) — the set of tasks ready to be worked right now.
- **The kanban board.** The same tasks, grouped into columns by status, so a whole project's progress reads as one glance across the board.
- **Inside a task.** Click any task to open its `task.md`: the objective at the top, the results an implementer wrote, and the review notes a reviewer left — including findings that were sent back for revision. This is the handoff surface agents and humans share; the [Task File reference](#/04-utility-skills/01-task-tree/01-task-file) is the full anatomy.

Every task path is a deep link. The address bar updates as you navigate, so you can bookmark or share a link straight to one task.

### The two trees

**A demo analysis, caught mid-flight.** A small, hypothetical research project: estimating whether firm size and price momentum predict next-month stock returns, on a fully simulated equity panel. The data are generated by a known process, so there is a built-in right answer; no real market data, tickers, or vendor records are involved. The tree is deliberately caught in the middle of its work: some tasks are `approved`, one is `implemented` and waiting on review, one is in `revise` after a reviewer caught that the portfolios were equal-weighted instead of value-weighted, and the write-up has not started. Most real projects spend most of their time in this state: partly done, with the frontier showing what to pick up next. It is the same analysis the [Quickstart](#/02-quickstart) walks you through building.

[Open the demo task tree →](demo-tree.html)

**superRA's own development tree.** superRA is built with superRA. The second export is the actual `superRA/` task tree of this repository: the work behind the workflow, the dashboard, the skill library, and this documentation site you are reading. It is larger and messier than the demo, because real work is. It shows how the task tree holds up on a non-trivial project.

### How these are built

Both exports are produced by the task-tree dashboard's standalone export — the same `generate` command documented in the [task-tree skill](skills/task-tree/SKILL.md) — which writes a single self-contained HTML file with everything inlined, no server required. The exports are rebuilt fresh in CI on every deploy and are never committed to the repository; only the task sources are. The exact build invocation lives in the deploy pipeline, which regenerates each tree with:

- **Demo tree:** `uv run --script skills/task-tree/scripts/plan_dashboard.py generate --plan-root docs/showcase-demo`
- **superRA's own tree:** `uv run --script skills/task-tree/scripts/plan_dashboard.py generate --plan-root superRA`

The export script is [plan_dashboard.py](skills/task-tree/scripts/plan_dashboard.py); the task-tree tooling that produces and reads these trees is documented in the [task-tree skill](skills/task-tree/SKILL.md).
