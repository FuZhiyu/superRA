## Objective

Replace the dashboard's single global root with a **per-worktree state cache** and a **per-request resolver**, then thread that resolver through every read route so each request renders the worktree it names. This is the foundation the SSE (02) and URL-routing (03) tasks build on; deliver it so the server behaves identically to today when no worktree is named, and correctly serves any other discovered worktree when one is.

**Deliverables:**

1. A `WorktreeState` dataclass holding the per-worktree render state currently spread across globals: `root_task`, `task_index`, `project_root`, `plan_root`. Replace direct use of the module singletons `_root_task` / `_task_index` / `_project_root` / `PLAN_ROOT` / `_current_worktree_path` ([plan_dashboard.py:65-82](../../../../../skills/task-system/scripts/plan_dashboard.py#L65)) with lookups into a module-level `_worktree_cache: dict[str, WorktreeState]` keyed by the resolved worktree id (the worktree directory basename per the root task's design decision 2).
2. A `resolve_worktree(request) -> WorktreeState` helper that reads the `?wt=<name>` query param, maps it to a discovered worktree via `_worktree_discovery.discover_worktrees()` ([_worktree_discovery.py](../../../../../skills/task-system/scripts/_worktree_discovery.py)), lazily builds and caches its `WorktreeState` on first use, and **falls back to the launch worktree when `?wt=` is absent or unknown**. Return a clear 404 only when `?wt=` names a value that is neither the launch worktree nor any discovered worktree.
3. Every read handler resolves the worktree from the request and renders from the resolved `WorktreeState` instead of the globals: `/`, `/tree`, `/nav`, `/nav/{path}`, `/node/{path}`, `/task/{path}`, `/dag`, `/kanban`, `/export`, `/files/{path}`, and the comment routes `/api/task/{path}/*` + `/api/comments/summary`. The render helpers ([plan_dashboard.py:329-395](../../../../../skills/task-system/scripts/plan_dashboard.py#L329)) already take explicit `task` / `project_root` args — pass the resolved values; do not read globals inside them.
4. Cache rebuild/invalidation hook for the watcher: expose a function that, given a worktree id, rebuilds or evicts that worktree's `WorktreeState` **and renders the broadcast fragments (`summary-updated`, `task:<path>`) from that `WorktreeState`** — i.e. the rebuild-and-render-fragment path operates on a resolved `WorktreeState`, not module globals. This is the part task 02 needs: task 02 owns the per-worktree watcher *lifecycle* and *delivery*, but it can only "keep the watcher body unchanged and just parameterize the root" if this task already moves `rebuild_tree` / `rebuild_task` / the summary+nav fragment rendering ([plan_dashboard.py:204-286](../../../../../skills/task-system/scripts/plan_dashboard.py#L204) call sites) onto `WorktreeState`. In this task, wire the hook to the existing single watcher so behavior is preserved; task 02 generalizes the watcher to per-worktree.
5. Preserve the launch path: the CLI `serve --root <path>` ([cli.py](../../../../../skills/task-system/scripts/cli.py)) and `lifespan()` startup ([plan_dashboard.py:406](../../../../../skills/task-system/scripts/plan_dashboard.py#L406)) seed the cache with the launch worktree as the default. `/export` standalone generation keeps rendering exactly one worktree.

**Scope boundary.** This task owns the state model and request resolution for **read** routes. It does NOT change SSE scoping or the watcher lifecycle (task 02) and does NOT add `?wt=` to the client URL or selector (task 03) — but it MUST make the server *accept and honor* a `?wt=` query param so 02 and 03 have a server to build against. The existing `POST /api/worktree/switch` may keep working against the launch/default worktree for now; its retirement is finalized in task 03.

## Results

The dashboard server now serves any discovered worktree on demand, resolved per request from a `?wt=<name>` query param, with the launch worktree as the default. The single global root is replaced by a per-worktree state cache; every read route resolves a `WorktreeState` and renders from it.

**State model.** A `WorktreeState` dataclass ([plan_dashboard.py](../../../../../skills/task-system/scripts/plan_dashboard.py)) holds the per-worktree render state (`wt_id`, `plan_root`, `project_root`, `root_task`, `task_index`) that previously lived in the `_root_task` / `_task_index` / `_project_root` / `PLAN_ROOT` singletons. A module-level `_worktree_cache: dict[str, WorktreeState]` is keyed by worktree id — the worktree directory basename (design decision 2), derived in `_worktree_id_for_plan_root()`. `_launch_wt_id` records the worktree the server started in. The legacy globals are retained as a compatibility view over the launch worktree's state (`_sync_default_globals()`), which the `/export` snapshot path and the standalone renderer still drive; `rebuild_tree()` reseeds the launch state and mirrors it into those globals, so the existing test suite and the CLI launch path are unchanged.

**Request resolution.** `resolve_worktree(request) -> WorktreeState` reads `?wt=`, returns the launch state when the param is absent or names the launch worktree, returns a cached state on a hit, and otherwise lazily builds the state from `_discovered_worktree_map()` (basename → plan root, built from `discover_worktrees()`, with longest-unique-suffix disambiguation on basename collisions per design decision 2). A `?wt=` that is neither the launch worktree nor any discovered worktree returns a 404; everything else falls back to the launch worktree. Every read route — `/`, `/tree`, `/nav`, `/nav/{path}`, `/node/{path}`, `/task/{path}`, `/dag`, `/kanban`, `/export`, `/files/{path}`, the comment routes `/api/task/{path}/*`, and `/api/comments/summary` — now resolves the worktree and renders from the resolved state. The render helpers (`_render_task_fragment`, `_render_task_node`, `_render_node_body`, `_render_summary`) take an explicit `project_root` / `root_task` and no longer read globals (they fall back to the legacy globals only when called with no argument, which is how the standalone renderer drives them).

**Watcher rebuild-and-render hook.** `rebuild_state_task(state, path)` and the new `_rebuild_and_broadcast(state, changes)` operate on a `WorktreeState`, rendering the `full-reload` / `summary-updated` / `task:<path>` fragments from that state rather than module globals. `rebuild_worktree_state(wt_id)` is the cache-invalidation hook task 02's per-worktree watcher will call: it re-walks the worktree's plan root and refreshes the cached state object in place (so any handler holding the reference sees the update). The existing single launch watcher (`_watch_plan_root`) is wired to this hook, preserving today's behavior; task 02 generalizes the watcher lifecycle to per-worktree.

**Scope honored.** SSE scoping and watcher lifecycle (task 02) and the `?wt=` client URL / selector (task 03) are untouched. `POST /api/worktree/switch` still works against the launch/default worktree (its retirement is task 03). `/export` resolves through the cache and exports exactly one worktree.

## Review Notes

Approved. The highest-risk seam — a resolved non-launch `WorktreeState` leaking into the standalone/`/export` render path that still reads module globals — is correct: `export_subtree` passes `state.plan_root` into `render_standalone_html` ([plan_dashboard.py:1129](../../../../../skills/task-system/scripts/plan_dashboard.py#L1129)), which re-walks that plan root and `_set_module_state`s the resolved worktree's tree before `_build_standalone_fragments` reads the globals ([plan_dashboard.py:1419-1437](../../../../../skills/task-system/scripts/plan_dashboard.py#L1419)), then restores the launch globals in the `finally`. Verified empirically: `/export?wt=<other>` renders the other worktree's tree only, and `_root_task` is restored to the launch tree afterward. Per-route resolution, the absent-`?wt=` → launch default, unknown-`?wt=` → 404, lazy-build-from-discovery, basename-collision suffix disambiguation, and the `WorktreeState`-parameterized `_rebuild_and_broadcast` / `rebuild_worktree_state` hook (task 02 can spin per-worktree watchers against it) were all read at source and exercised against the live module. The two items below are non-blocking advisories for tasks 02/03.

1. **MINOR** — Launch-basename collision creates a duplicate cache entry. `resolve_worktree` ([plan_dashboard.py:565](../../../../../skills/task-system/scripts/plan_dashboard.py#L565)) short-circuits `?wt=<launch basename>` to the launch state before consulting `_discovered_worktree_map()`, while the launch worktree is cached under its bare basename ([plan_dashboard.py:168-169](../../../../../skills/task-system/scripts/plan_dashboard.py#L168)). When the launch basename collides with another discovered worktree, the map drops the bare key and registers disambiguated suffixes (`p1/samebase`, `p2/samebase`); a request for the launch worktree's own *disambiguated* suffix key then misses the short-circuit, misses the cache, and lazily builds a second `WorktreeState` for the same worktree under the suffix key. Harmless duplication (correct content either way), but task 03's URL/selector work should pick one canonical id for the launch worktree so its links don't double-cache it.

2. **MINOR** — No unit test pins the lazy-build-from-discovery or basename-collision paths. `test_unknown_wt_returns_404` relies on the tmp trees not being real git worktrees so discovery returns nothing; the positive "unknown-to-cache but discoverable `?wt=` builds and renders that worktree" path and the longest-unique-suffix collision logic in `_discovered_worktree_map` ([plan_dashboard.py:518-553](../../../../../skills/task-system/scripts/plan_dashboard.py#L518)) are covered only by the live-server check, not a regression test. Task 04 (cross-worktree integration tests) is the right place to add a `discover_worktrees`-monkeypatched case so a future change to the discovery/collision logic goes red.
