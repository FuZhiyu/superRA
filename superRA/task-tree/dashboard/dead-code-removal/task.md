## Objective

Remove the unreachable pre-master-detail rendering subsystem so the dashboard has exactly one row-rendering path (`nav_node.html`).

- Delete the `GET /tree` and `GET /task/{path}` routes, the `_render_task_fragment` / `_render_task_node` helpers, the `task_node.html` and `task_children.html` templates, and the client-side cluster `toggleTask` + its `/task/` lazy-load + the `:scope > .task-body` branch of `updateTreeCommentBadges`.
- After removal, no duplicated row markup remains: the row + approved-count loop lives only in `nav_node.html`.
- The standalone export must not regress — verify (not assume) that `_build_standalone_fragments` and the client fetch shim never reference the removed routes or templates.
- Validation: full dashboard and task-tree suites green; a grep for the removed names finds nothing outside test history; a live serve renders sidebar, active card, kanban, and DAG views, and a fresh export opens with rendering intact.

## Planner Guidance

Evidence the subsystem is dead (2026-07-19 design review): the shipped page renders the sidebar via `/nav` (`nav_node.html`) and the active card via `/node/` (`node_body.html`); `task_node` markup is emitted only by `/tree`, which returns a bare fragment with no script, so its `onclick="toggleTask(this)"` handlers are non-functional there too. Client cluster at [base.html:2553-2596](../../../../skills/task-tree/scripts/templates/base.html#L2553-L2596); dead badge branch at base.html:4856-4857 (nav rows have no `.task-body`); duplicated row markup [nav_node.html:31-45](../../../../skills/task-tree/scripts/templates/nav_node.html#L31-L45) vs [task_node.html:87-101](../../../../skills/task-tree/scripts/templates/task_node.html#L87-L101); routes at [plan_dashboard.py:1003-1023](../../../../skills/task-tree/scripts/plan_dashboard.py#L1003-L1023) and 1383-1391.

`test_dashboard.py` pins some of these surfaces — update or remove those tests as part of this task, and say in `## Results` which tests were retired and why.

## Results

Removed the entire dead giant-tree subsystem; the dashboard now has exactly one row-rendering path (`nav_node.html`) and one body-rendering path (`node_body.html` via `render_task_body`).

**Backend** ([plan_dashboard.py](../../../../skills/task-tree/scripts/plan_dashboard.py)): deleted `_render_task_fragment`, `_render_task_node`, the `GET /tree` route, and the `GET /task/{path}` route. `_task_depth` survives — it's still used by `_render_nav_node`. Updated stale docstrings/comments that referenced the removed route/helper names (the `/nav` docstring, the comment-route ambiguity note, the `/nav/{path}` ordering comment).

**Templates**: deleted [task_children.html](../../../../skills/task-tree/scripts/templates/task_children.html) outright (only consumer was the removed `_render_task_fragment`). `task_node.html` contained two macros: `render_task_node` (the dead row+body wrapper, duplicate of `nav_node.html`'s row) and `render_task_body` (the live body-only macro `node_body.html` depends on — confirmed via grep, `render_task_body` is not duplicated markup, just the body half). Deleted `render_task_node` and moved `render_task_body` into a new file, [task_body.html](../../../../skills/task-tree/scripts/templates/task_body.html), since a file named `task_node.html` containing only a body macro would misdescribe its contents; updated the one import site ([node_body.html](../../../../skills/task-tree/scripts/templates/node_body.html)) accordingly. Cleaned up stale cross-references in [nav_children.html](../../../../skills/task-tree/scripts/templates/nav_children.html) and [nav_node.html](../../../../skills/task-tree/scripts/templates/nav_node.html) that pointed at the deleted files.

**Client JS** ([base.html](../../../../skills/task-tree/scripts/templates/base.html)): deleted `toggleTask` (its only caller was `task_node.html`'s row `onclick`, confirmed by grep) and its `/task/` htmx lazy-load call. Removed the `:scope > .task-body` check in `updateTreeCommentBadges` — confirmed dead because `.task-body` is only emitted by the now-deleted `render_task_node` wrapper; neither the sidebar rows (`nav_node.html`) nor the active-node card body (raw `node_body.html` output) ever carry that class, so the branch's `body && ...` guard was always false. Left the `.task-body`/`.task-body.open`/`.task-body.uncapped` CSS rules alone — the task's bullet list scopes JS removal explicitly and doesn't call for a CSS sweep; flagging here as now-unreferenced CSS a future frontend pass could prune. `uncapAfterTransition`/`recapForCollapse` are kept — still used by `toggleSection` and `patchCardBadgeWhenReady`.

**Standalone export**: verified (not assumed) — `_build_standalone_fragments` only ever built `/nav`, `/node/<path>`, `/dag?root=<path>`, `/nav/<path>`, and `/kanban` fragments; the client's `standaloneFetch` resolves purely by map lookup with no hardcoded route names. No changes needed there.

**Tests retired/updated:**
- `test_dashboard.py`: removed `test_get_task_returns_children_fragment`, `test_get_task_valid_path`, `test_get_task_nonexistent_returns_404` (tested the deleted `/task/{path}` route directly; equivalent coverage for the surviving `/nav/{path}` route already exists in `test_nav_path_returns_deep_children_body_free` / `test_nav_path_missing_404`). Removed `test_node_sections_match_full_node`, which asserted byte-for-byte parity between `/node/{path}` and `/task/`'s duplicate section markup — with the duplicate path gone there is nothing left to compare against; `render_task_body` is now the sole section-markup source, so the parity claim is vacuously true and not worth a test. Dropped `/tree` from `test_existing_routes_unaffected`'s route list. Converted `test_render_task_node_has_sse_swap` / `test_render_task_node_has_badge` / `test_postponed_renders_badge_and_status` to call `_render_nav_node` instead (same behavioral assertions — sse-swap attr, badge class, status — now exercised through the live render path). Converted `test_script_payload_escapes_closing_script_tag` to call `_render_node_body` instead of `_render_task_node`, since the `</script>` escaping behavior it tests lives in the body macro, not the row macro.
- `tests/test_state_preservation.py`: removed `TestTreeRoute` (5 tests wholly scoped to the deleted `/tree` route). Renamed `TestRenderTaskNodeDepth` to `TestRenderNavNodeDepth` and repointed its calls at `_render_nav_node`; dropped its `test_depth_inferred_from_path` as a pure duplicate of `TestTaskDepth::test_*` below it, and dropped the `needsLoad` assertion from `test_children_lazy_at_high_depth` since `nav_node.html` (unlike the deleted `task_node.html`) never emits a server-side `needsLoad` script — lazy children are flagged client-side by `markLazyNodes()`. Fixed the three `_render_task_node` calls inside `TestWatcherDecisionLogic._run_watcher_logic`'s hand-rolled broadcast simulation to call `_render_nav_node`, matching what production `_rebuild_and_broadcast` actually calls (confirmed at [plan_dashboard.py:424](../../../../skills/task-tree/scripts/plan_dashboard.py#L424)); also corrected two stale `_watch_worktree()` docstring/comment references to the real function name, `_rebuild_and_broadcast`. Removed the now-orphaned `client` fixture (its only consumer was `TestTreeRoute`).
  - Observed but not fixed (out of this task's scope): `_run_watcher_logic`'s simulation of *structural* changes (add/delete) still models a per-parent `task:` broadcast, while production `_rebuild_and_broadcast` does a full-reload for structural changes instead — a pre-existing drift between this hand-rolled harness and the real function it claims to mirror, independent of this task's dead-code removal. `test_dashboard.py::test_content_edit_under_a_is_scoped_to_a` already exercises the real `_rebuild_and_broadcast` directly for content-only changes, so this drift affects test-harness fidelity, not coverage of correct behavior.
