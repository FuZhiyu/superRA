## Objective

The dashboard server fails loudly and its connection bookkeeping stays truthful.

- A `task.md` that fails to parse during a watcher rebuild is surfaced — at minimum a server-side log line and a visible error state on that node in the client — instead of silently serving the last-good parse with no signal.
- A slow SSE client whose queue overflows has its connection ended rather than being silently unsubscribed, restoring the invariant the idle monitor depends on: registered queues == open connections (today the server can idle-exit under a live but stalled tab).
- The root-or-children inline template string that is duplicated verbatim in three places becomes one shared precompiled template, and the per-call `env.from_string` render helpers use precompiled templates.
- Stale dispatch-context comments ("task 01", "task 02", "created by another agent") are removed from the module.
- Validation: a test covers parse-error surfacing (broken fixture task.md → error state, not silent staleness); a test covers queue-overflow ending the SSE stream and the idle count staying consistent; both suites green.

## Planner Guidance

Findings (2026-07-19 review, [plan_dashboard.py](../../../../skills/task-tree/scripts/plan_dashboard.py)): the silent swallow is `except Exception: return state.task_index.get(task_path), False` at 249-251; the eviction desync is `_broadcast` discarding a full queue at 333-340 while the client's generator keeps the connection open (its `finally` cleanup at 1090-1096 never runs), so `_open_connection_count` undercounts; duplicated template strings at 1012-1021 / 1040-1049 / 1494-1503; `from_string` per call (bypasses the template cache) at 791 and 807; stale comments at 121, 349, 1060 and nearby.

For ending an overflowed client: you cannot enqueue a sentinel into a full queue — consider draining-and-replacing with a close sentinel, tracking evicted queues that the generator checks each loop, or cancelling the response task. Design is the implementer's; the binding requirement is only that the connection ends and the count stays truthful.

## Results

All four fixes landed in [plan_dashboard.py](../../../../skills/task-tree/scripts/plan_dashboard.py), plus a supporting `Task.parse_error` field in [_task_io.py:107](../../../../skills/task-tree/scripts/_task_io.py#L107) and a badge in [nav_node.html](../../../../skills/task-tree/scripts/templates/nav_node.html) / [base.html](../../../../skills/task-tree/scripts/templates/base.html).

- **Parse-error surfacing** — [rebuild_state_task](../../../../skills/task-tree/scripts/plan_dashboard.py#L213), `except Exception as exc:` at [plan_dashboard.py:228-241](../../../../skills/task-tree/scripts/plan_dashboard.py#L228-L241): on a re-parse failure the watcher now `print`s a `[dashboard] failed to parse …` line to stderr (captured by the background server's log file), then stamps the last-good `Task` with `parse_error` via `dataclasses.replace` and writes it back into both the flat index and the tree node — so the error state also shows up on a fresh page load, not just the live SSE fragment. [nav_node.html](../../../../skills/task-tree/scripts/templates/nav_node.html) renders a `parse error` badge (`badge-error`, reusing the existing `--st-rev`/`--st-rev-t` error-color tokens per [base.html](../../../../skills/task-tree/scripts/templates/base.html)) and a `data-parse-error="true"` attribute whenever `task.parse_error` is set.
- **Truthful SSE bookkeeping** — [_broadcast](../../../../skills/task-tree/scripts/plan_dashboard.py#L318) no longer discards a full queue from `_worktree_clients` itself (that undercounted `_open_connection_count()` while the stalled client's response stayed open with nothing left to wake it). It now drops the queue's oldest buffered message and enqueues a `_QUEUE_CLOSE` sentinel ([plan_dashboard.py:315](../../../../skills/task-tree/scripts/plan_dashboard.py#L315)); the `/events` generator ends the stream when it reads that sentinel ([plan_dashboard.py:1079](../../../../skills/task-tree/scripts/plan_dashboard.py#L1079)), so removal from `_worktree_clients` happens exactly once, in the generator's own `finally`, the same path a normal disconnect takes. Registered queues == open connections holds even for a stalled tab.
- **Template consolidation** — the root-or-children snippet was duplicated in two remaining call sites (`/nav`'s route handler and `_build_standalone_fragments`'s export path; the third occurrence the Planner Guidance cited was already removed by the `dead-code-removal` task's `/tree` deletion, per the dispatch note). Both now call a shared `_render_root_or_children` ([plan_dashboard.py:818](../../../../skills/task-tree/scripts/plan_dashboard.py#L818)) backed by one lazily-compiled, cached `Template` ([_get_root_or_children_template](../../../../skills/task-tree/scripts/plan_dashboard.py#L810)). The single-node nav snippet (`_render_nav_node`) is likewise now backed by a cached `Template` ([_get_nav_node_snippet_template](../../../../skills/task-tree/scripts/plan_dashboard.py#L802)) instead of a fresh `env.from_string` per call.
- **Stale comments removed** — the "task 01" / "task 02" / "created by another agent" dispatch-context references are gone from the module (verified via `grep -n "task 01\|task 02\|another agent"` returning no hits); the surrounding prose was reworded in place to describe the current mechanism instead of the dispatch that introduced it.
