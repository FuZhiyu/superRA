## Objective

> The demo tree and the superRA dev-tree export described below were later retired by [14-canonical-showcase](../14-canonical-showcase/task.md), which leaves the real [showcase-analysis](../../showcase-analysis/task.md) study as the sole showcase tree. This task's record stands as the history of what shipped at the time.

Produce the showcase that lets a visitor *see* the task tree instead of reading about it:

- **A curated demo task tree** — a small, realistic, public-safe hypothetical research analysis (a handful of tasks with objectives, results, review notes, a figure, statuses mid-flight) — committed as a source tree and exported with the standard (non-doc-mode) standalone export so the full task-tracker UI shows: status pills, rollup, DAG, kanban.
- **superRA's own development tree** — the real `superRA/` tree of this repo, exported the same way, as the existence proof.
- Both exports wired into the site per the IA (linked pages or embeds), with a short framing page explaining what the visitor is looking at and how it maps to the concepts section.
- A documented regeneration command for each export so `08-deploy` can rebuild them in CI; exports themselves are not committed.

Validation: open each exported file and verify the trees render with full chrome, deep links work, and the demo tree contains no real personal or project data.

## Results

The showcase ships three pieces: a curated demo task tree (committed source), the framing page, and the documented regeneration commands for both exports. The exports themselves are CI-built, never committed.

### Curated demo task tree

A small, public-safe hypothetical analysis at [docs/showcase-demo/](../../../docs/showcase-demo/): "Size and Momentum Effects in Simulated Equity Returns" — does firm size and price momentum predict next-month returns on a *fully simulated* equity panel. The data-generating process plants a known size and momentum premium, so there is a built-in right answer; no real tickers, vendor records, or personal data appear anywhere in the tree (scanned clean for personal identifiers). The story is the same toy analysis [04-quickstart-tutorial](../04-quickstart-tutorial/task.md) is slated to walk through; coordination note in §Coordination below.

Seven nodes, with statuses caught mid-flight so the export exercises the full task-tracker chrome:

| Node | Status | Exercises |
|---|---|---|
| root | `revise` (rolled up) | rollup — a `revise` child pulls the parent un-finished |
| 01-simulate-data | `approved` | results + closed review notes (two findings implemented) |
| 02-construct-factors | `approved` | results + closed review notes |
| 03-size-sort-returns | `implemented` | results + an embedded figure (committed under `attachments/`) |
| 04-momentum-sort-returns | `revise` | open review notes with a live MAJOR finding |
| 05-double-sort-robustness | `in-progress` | partial results, blocked on the upstream `revise` |
| 06-writeup | `not-started` | a frontier task waiting on its dependency |

The dependency edges form a diamond (`02` fans out to `03`/`04`, both feed `05`), so the DAG view shows non-trivial structure rather than a chain. Statuses span `approved` / `implemented` / `revise` / `in-progress` / `not-started`, so status pills and the kanban board show every color.

The figure on `03-size-sort-returns` is committed at [docs/showcase-demo/03-size-sort-returns/attachments/size_decile_returns.png](../../../docs/showcase-demo/03-size-sort-returns/attachments/size_decile_returns.png) and regenerated by its sibling [make_figure.py](../../../docs/showcase-demo/03-size-sort-returns/make_figure.py) (PEP 723 script, `uv run --script`); the export base64-inlines it.

### Framing page

[docs/site/06-showcase/task.md](../../../docs/site/06-showcase/task.md) — the `06-showcase/` node from the IA sitemap. It tells a visitor what they are looking at (status pills, rollup, DAG, kanban, the review conversation inside a task), maps each element to its owning concept/reference page via `#/` hash links, introduces the two trees, and documents how they are built. All six cross-page hash links and both repo-file links were verified to resolve against the scaffolded sitemap and real repo files. Authored Draft-mode for the researcher audience; links to the authoritative `task-tree` skill and `See Your Work` how-to rather than paraphrasing them.

### Regeneration commands (for 08-deploy)

Both exports use the standard full-chrome standalone export (non-doc-mode), self-contained single HTML, run fresh in CI and never committed:

- **Demo tree:** `uv run --script skills/task-tree/scripts/plan_dashboard.py generate --plan-root docs/showcase-demo --output <site>/demo-tree.html`
- **superRA's own tree:** `uv run --script skills/task-tree/scripts/plan_dashboard.py generate --plan-root superRA --output <site>/superra-dev-tree.html`

The framing page links to the two exports by relative filename (`demo-tree.html`, `superra-dev-tree.html`); `08-deploy` owns the final output paths and must place the two export files alongside the site export so those links resolve. Final filenames/placement are a deploy decision — if `08-deploy` chooses different names, the two relative links in the framing page body update with them.
