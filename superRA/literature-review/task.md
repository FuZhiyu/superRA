## Objective

Own the literature-review domain vertical. `skills/literature-review/` teaches the search -> screen -> synthesize discipline for **economics/finance** literature reviews, and for now its orchestration assumes a **superRA task-tree review** rather than a standalone folder-ledger workflow. The review root task is the researcher-facing progress page; review-agent dispatches remain visible as task-tree work, while pre-screen paper records live in a non-git candidate-paper store until they are included or explicitly escalated. It composes existing skills rather than restating them — `zotero-paper-reader` (save to Zotero, fetch PDFs, cite/BibTeX), `mistral-pdf-to-markdown` (OCR a curated shortlist), and `writing` (any prose) — and bundles one citation-graph + metadata client plus a candidate materialization/dedup mechanism. The vertical is wired into the skill-discovery surfaces and protected by a skill-trigger test, mirroring `slide-design-vertical`.

The deliverable a review produces is a **curated, provenance-tracked, extracted collection** (screened set + per-paper justification + quote-grounded extraction + convergence judgment), not a drafted prose review. Synthesis, classification, and gap analysis are **bespoke to each review** and out of scope for the vertical — the skill delivers the organized evidence, and any downstream map is authored per project.

### Context

This is skill-creation work in the superRA repo. Load `skill-creator` before editing any `skills/*/SKILL.md`; keep instructions minimal and behavior-shaping; update the inventory surfaces when adding a skill. The repo is **public** — tests and examples must never carry personal library data (real group names/ids, item keys, titles, counts, query results); use placeholders and hypothetical examples.

The manual run's mechanism is the spine; the research adds automation (citation APIs), anti-hallucination grounding, and econ/finance corpus discipline.

### Conventions

**Discovery is multi-modal — no single search tool.** Semantic Scholar and other citation-graph indexes lag on the newest working papers and have coverage holes, so front-line discovery **leads with web search** — the agent's WebSearch/WebFetch plus site-scoped queries (`site:ssrn.com`, `site:nber.org`, RePEc/IDEAS, arXiv q-fin, author pages), run as several blind lenses and deduped. Semantic Scholar's primary role is the **citation graph** (backward `/references`, forward `/citations`) and semantic-similarity search over indexed work — not sole front-line search. **Crossref** is the authoritative published-version-of-record resolver by DOI/title (covers NBER). arXiv supplies q-fin/econ preprint abstracts. Google Scholar has the broadest WP coverage but no API and blocks scraping — approximate it with WebSearch + site-scoped queries. **No OpenAlex** (data quality). **OpenCitations** is the keyless forward-citation fallback when S2 is rate-limited — DOI→DOI edges from Crossref open references (I4OC), distinct from OpenAlex and without its quality problem; forward citations of a working paper must be unioned across its version DOIs (NBER `10.3386/*`, SSRN `10.2139/ssrn.*`, journal). **No Docker / translation-server** (setup friction) — the Zotero item is built from the verbatim record via a small mapper.

**No API keys required.** Semantic Scholar, Crossref, and arXiv are all keyless — S2's anonymous pool suffices because discovery is web-first, so S2 is called only for citation-graph traversal on already-screened papers (low volume). **S2 is optional:** the loop runs on web search + Crossref alone, with S2 added only to automate backward/forward citation snowballing, and OpenCitations (keyless) as the forward fallback when S2 throttles. Because parallel discovery/screening work shares one IP, the client coordinates its API calls through shared on-disk state (a filesystem rate gate + response cache) so concurrent agents stay inside backend rate limits — see [citation-client-resilience](citation-client-resilience/task.md). The only keys anywhere are Mistral (OCR shortlist) and — for the cloud Zotero write path only — a Zotero write key; the local connector needs none.

**Metadata policy — never agent-authored.** Bibliographic fields are queried and stored **verbatim**; an agent never types them. Metadata is always taken from the **published version of record** (Crossref DOI). SSRN working papers usually lack a Crossref DOI → discover/resolve them via S2/web, and adopt the published Crossref metadata once a published version exists. When a paper is in **no structured index** (a brand-new WP found via web search), take metadata from the publisher/repository page's embedded **`citation_*` / Dublin Core meta tags** (the structured fields Zotero's web translators read) — still verbatim from the source page, never agent-composed.

**PDF policy.** Fetch the PDF best-effort. On a paywall, fall back to the **latest freely-available version** (preprint/WP) and **flag the divergence** in the ledger (e.g. metadata = published JF 2024; PDF = 2021 SSRN WP). WP and published versions differ materially in economics, so the flag is required. Store fetched PDFs and OCR output in the project's durable artifact store — Zotero attachments when the add policy targets Zotero, else the project's local papers/artifacts convention — and record relative paths from the candidate or promoted paper record. Fetch and OCR are **idempotent**: reuse an existing stored copy rather than downloading or (billed) OCR-converting the same paper twice.

**Anti-hallucination.** Resolve every DOI against Crossref/S2 before trusting a cite (64% of fabricated cites carry a real-but-unrelated DOI). Ground extracted claims in verbatim quotes. Distinguish a true null from "not reported."

**Researcher-facing tree.** The review root task is the live dashboard page. It records the research question, criteria/schema, corpus counts, the active review frontier, unresolved dedup conflicts, and saturation evidence, with links to review tasks and promoted paper cards. The tree should not create a task for every pre-screen candidate.

**Candidate-paper store before screening.** Pre-screen candidates live outside git in the project's convention for local review data (for example `Notes/literature-review/<review>/candidate-papers/`). Each candidate is a task-shaped folder with a `task.md` using the same body template as an eventual paper card, so promotion does not require schema conversion. The materializer creates or updates these folders from discovery metadata, names them consistently, and merges newly surfaced provenance into the existing folder when dedup identifies the same paper. It does not use JSON/JSONL as authoritative state.

**Promotion into the task tree.** Only included papers, clearly central escalation cases, and final curated papers become `superRA/<review>/papers/<paper-key>/task.md` cards. Promotion **moves** the task-shaped candidate folder into the review tree (not a copy) and repoints any candidate-store links that pointed at its old path, so a promoted paper never leaves a stale duplicate behind. Excluded and merely pending candidates remain auditable in the local candidate-paper store, not as dashboard task cards.

**Read-once, one role — the agent that opens a paper finishes it.** A single review-agent role does the full per-paper work; there is no discovery/screening/extraction role boundary. A dispatch assigns a frontier — a seed to expand, a cluster of pending candidates, or a web lens — plus loose bounds: recommended citation hops (typically 2), an extraction authorization, and a soft paper budget. Within bounds the agent's on-the-ground judgment decides what to open: how the text in hand cites a paper is the primary signal for whether it is worth pursuing, so edge triage happens while reading, with quoted citation context in the stub's provenance; bulk edge lists with no read context are triaged from metadata and stubbed by priority. One session does everything its reads enable — screening decision, extraction when authorized and warranted, lead harvest — and beyond bounds the agent stubs and reports rather than chasing. Before opening any source the agent claims it atomically in the shared store; a paper already claimed or decided is adopted, not re-read. Records are completed paper-by-paper (write-as-you-go), so an exhausted context loses nothing — the remaining frontier persists as stubs. Each record's `## Reading Notes` is an append-only read log for the resume cases (deferred extraction, escalation, late centrality upgrade); a resumed session adopts and re-verifies facts already grounded with a quote instead of re-deriving them.

**Extraction shape is a researcher-elicited hybrid, not a fixed matrix.** Setup elicits which facts must be comparable across every included paper (may be empty) — those become shared matrix columns; everything else is a free-form grounded narrative note per paper. An empty comparison-column set collapses the shape to narrative-only. The grounding invariants (DOI resolution, quote-per-claim with location, extract-then-verify, null vs "not reported") are mandatory regardless of shape.

**Naming and dedup.** A simple CLI materializer computes the canonical candidate/paper folder name from verbatim metadata and identifiers: prefer published-version metadata when a DOI/Crossref record exists, otherwise the best working-paper source. Names should be human-readable and stable enough for navigation, e.g. `<firstauthor>-<venue-or-wp>-<year>-<short-title>` with a deterministic suffix only for collisions. Dedup is mechanism-owned: DOI/arXiv/S2/title-year matching updates an existing candidate folder when the identity is clear and surfaces only flagged conflicts for orchestrator judgment. Agents may split or override a merge when they determine two records are actually different papers.

**Read to the depth the paper warrants, in one session.** The binary in/out call resolves from metadata / abstract / intro — no bulk OCR to triage. When a paper is central and clearly included, the same session reads its related-work / citation discussion (cheaply-available text; not an OCR trigger) to steer frontier priorities, recording `read_depth`, and runs extraction immediately when the dispatch authorized it — OCR via `mistral-pdf-to-markdown` (idempotent) fires only at extraction depth, never for triage. Read to the depth the decisions need, no deeper; an included paper without extraction authorization exits the session flagged `extraction-pending`.

**Econ/finance discipline.** Working-paper-first coverage (SSRN / NBER / RePEc), version-tracking, JEL codes as a scope/audit facet, and weighting by outlet tier + identification strategy — not crawlability or raw citation count.

**Snowball + convergence — written state, local judgment, mechanism-owned dedup.** Backward citation BFS from seeds plus a forward multi-lens web sweep writes all surfaced candidates into the local task-shaped candidate-paper store. Review agents judge frontier value, decide membership, and harvest citation leads within their assigned frontier and bounds; the materializer owns ordinary dedup, claim coordination, and canonical folder creation/update; the orchestrator resolves only flagged identity conflicts and owns synthesis, review, researcher communication, and the final saturation judgment. Stop when the written state shows no unscreened candidate backlog, no unexpanded included frontier that matters to scope, and the latest backward/forward discovery pass adds no new in-scope paper; record the stopping judgment and deliberate non-expansions on the review root and saturation-audit task.

**Provenance carries a retrieval trace.** Beyond discovery lineage (`discovered_via`, `bfs_depth`), each entry records how to refetch the artifact — `ids`, `landing_url`, `pdf_url` + `access`, `pdf_path`, `md_path`, `fetched_at` (the version-divergence flag lives here). A surfaced candidate carries the normalized record the discovering agent already held, so the next round's agent starts from a handle, not a cold search.

**Dispatch is a cost-control strategy, not dashboard structure.** Dispatches are frontier assignments with bounds, sized by expected depth — a web-lens sweep is wide and shallow, a seed expansion narrow and deep. They run concurrently as ordinary parallel Agent calls without worktree isolation, because the only shared mutable state is the non-git candidate store and the client cache, both concurrency-safe, and claim-on-open prevents two dispatches processing the same paper. Batch labels and tier language should not become primary task-tree objects. The dashboard remains review-root progress plus navigable promoted paper cards.

**Skill file layout — partition by loader.** The skill is loaded by the **main agent** for interactive setup, written-state orchestration, candidate materialization/dedup supervision, synthesis, review, and researcher communication; review agents load the review-agent reference plus `econ-corpus.md`, adding `grounding-and-extraction.md` only at extraction depth. Orchestration lives in a main-agent-only reference; SKILL.md carries shared framing, references map, composed-skills routing, and the candidate/paper record schema; per-paper domain discipline lives in the references the implementer loads. See CLAUDE.md §Minimal, Targeted Instructions.
