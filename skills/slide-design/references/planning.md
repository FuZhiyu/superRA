# Slide-Design Planning

Use this reference in PLAN phase when slide work enters the superRA workflow.

## Audience-Context Inventory

Before task decomposition, record the audience-context inventory in the governing ancestor task's objective:

- **Audience:** venue, likely fields, seniority, and whether they know the paper/project.
- **Shared context:** concepts, methods, data, notation, and institutions the audience can be expected to know without setup.
- **Missing context:** concepts, notation, mechanisms, empirical design details, or institutional facts that must be introduced before they can support a claim.
- **Divergent priors:** phrases or claims likely to trigger different interpretations across fields.
- **Talk objective:** the one thing the audience should understand or believe by the end.
- **Main-vs-backup policy:** which rigor, robustness, expert objections, and derivations belong off the main path.

The inventory is a planning artifact, not prose for the slides.

## Deck Structure

Plan the deck around the audience's context state, not around the paper's section order.

- Start with the intended takeaway or question before details that need that frame.
- Insert context-setting slides before sections where familiar terms can mislead the audience.
- Use roadmap returns when the audience needs to reset attention or locate the current section.
- Treat model, data, and empirical-design slides as communication devices: explain why the audience needs the object before showing its mechanics.

## Task Decomposition

Good slide-design tasks are organized by communication unit:

- audience/context inventory and deck route;
- title/takeaway pass;
- dense technical-slide simplification;
- overlay and visual-cue pass;
- main/backup split;
- layout/build pass.

Each task should name the intended audience state before and after the task. A task that only says "polish slides" is underspecified unless it points to a concrete deck section and communication objective.
